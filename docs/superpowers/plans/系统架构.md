### 系统：六阶段、十一对策架构

```
[原始表格] ➔ 1.解析与双轨对齐 ➔ 2.【Gate 1: 表格结构确认】 ➔ 3.动态公式推导 ➔ 4.混合异常检测 ➔ 5.摘要压缩 ➔ 6.AI分析与图表生成
```

------

### 第一阶段：表格重构与“双轨”对齐（数据输入层）

此阶段引入“维度与度量分离”以及“早期人工介入（HITL）”机制，防止级联失效。

#### 步骤 1.1：版面分析与行列定位（同前）

- **功能：** 定位表格并提取 OCR 单元格物理边界。

#### 步骤 1.2：双轨制单元格填充（修复漏洞 1）

- **功能：** 区分处理合并单元格。**维度单元格（表头/文本分类）：** 采用 **Padding 机制**，向下/向右复制填充（如：将“2023年”合并项复制到所有子列）。**数值单元格（数据主体）：** 采用 **NaN 留空或比例拆分机制**。在转换为 Pandas DataFrame 时，仅在合并区域的左上角单元格保留数值，其余合并区域填充为 NaN（或根据业务设定按行列比例分摊），防止后续求和（Sum）等聚合计算时数据被翻倍。
- **输出：** 结构对齐、数值不失真的二维临时矩阵。

#### 步骤 1.3：【Gate 1】主页面交互确认（修复漏洞 5）

- **功能：** **将人类反馈（HITL）大幅前置**。在数据进入任何计算或 AI 分析之前，在前端弹出一个简易的“网格预览与纠错”界面。展示 OCR 识别出的行列划分线（用红色线框标记在原图上）。允许用户点击鼠标进行“合并/拆分/微调行列线”的简单纠错。
- **量化指标：** 用户确认后，数据结构准确率达到 **100%**，方可放行进入后续自动化流水线。

------



### 第二阶段：语义理解与动态关系推导（结构理解层）

引入“大模型动态推导公式”机制，解决硬编码的盲区。

#### 步骤 2.1：多级表头关系提取

- 
- **功能：** 解析表头，确定时间、地区、业务指标等维度。

#### 步骤 2.2：AI 动态公式发现器（修复漏洞 3）

- 
- **功能：**将表头结构及少量样本行发送给轻量级大模型。提示词约束大模型仅完成一项任务：**识别表格中的逻辑关系和隐式公式**，并输出结构化 JSON 表达式。*LLM 输出示例：*codeJSON`[  {"formula": "列C = 列A - 列B", "description": "营业利润 = 营业收入 - 营业成本"},  {"formula": "行10 = sum(行3:行9)", "description": "总计 = 各分项之和"} ]`
- **输出：** 该表格专属的校验公式库（动态生成，无需人工硬编码）。

------



### 第三阶段：高精度校验与数据清洗（计算与异常检测层）

此阶段解决校验规则失效、小样本误报和数据量过载问题。

#### 步骤 3.1：确定性动态公式校验

- **功能：** Python 校验引擎读取步骤 2.2 生成的 JSON 公式库，并转化为 Pandas 的 eval 表达式，在 DataFrame 上运行。
- **输出：** 运算不一致的“计算错误点”清单（准确率由确定性代码保证）。

#### 3.2：混合策略异常检测（修复漏洞 4）

- **功能：** 根据有效数据行数 `NN`，自动切换算法策略。
  - **大样本模式（**`**`N≥30N≥30`**`**）：** 启动统计学模型（如 IQR 或孤立森林），辅以业务规则（阈值）。
  - **小样本模式（**`**`N<30N<30`**`**）：** 关闭机器学习模型，改用**业务逻辑判定器**（如：增长率超 ±50%、连续3期单调递减、数值与零的关系等硬性业务规则）。
- **输出：** 剔除高误报后的“业务异常点”清单。

#### 3.3：上下文摘要压缩引擎（修复漏洞 2）

- **功能：** 拒绝将全量明细表直接喂给大模型。通过 Python 计算生成两份摘要数据：**统计摘要（Summary Table）：** 均值、极值、中位数、环比波动率等（通常只有几行到十几行）。**异常明细切片（Anomaly Slices）：** 仅提取步骤 3.1 和 3.2 判定为错误的行和单元格，并附带错误原因说明。
- **输出：** 压缩后的上下文包（相比全量表格，体积压缩 80% 以上）。

------



### 第四阶段：AI 深度业务分析（AI 认知层）

此时，大模型接收到的是“高度精简、结构明确、包含预诊断结果”的数据包。

#### 步骤 4.1：精炼上下文分析

- **功能：** 大模型基于步骤 3.3 的“统计摘要 + 异常切片”进行推理，避免了 "Lost in the Middle" 现象。由于大模型无需处理繁琐的加减乘除计算（计算工作已在第三阶段由 Python 完成），其注意力可以全部集中于**解释异常、归纳趋势和提供业务决策建议**。
- **输出：** 结构化的分析洞察 JSON（包含结论、严重程度分级、对应的单元格定位等）。

------



### 第五、六阶段：智能图表与系统自愈（同前）

利用前面阶段输出的清洁数据，进行高稳定性的图表推荐和渲染。



### 总结：打补丁后的系统健壮性提升

| 漏洞              | 风险后果               | 补丁方案                                               | 修复后的系统表现                       |
| ----------------- | ---------------------- | ------------------------------------------------------ | -------------------------------------- |
| **1. 单元格填充** | 导致聚合计算数值翻倍   | **双轨制**：维度复制，数值 NaN 或拆分                  | 计算准确率不受合并单元格干扰           |
| **2. Token 溢出** | 触发 LLM 遗忘、高成本  | **摘要压缩**：仅发送统计摘要和异常切片                 | Token 消耗降低 80%，分析聚焦率大幅提升 |
| **3. 硬编码盲区** | 遇到陌生表头校验失效   | **AI 公式推导 ➔ Python 校验**                          | 系统具备跨行业的自适应财务校验能力     |
| **4. 小样本误报** | 小表格导致统计算法误报 | **混合策略**：`N≥30N≥30` 运行模型，`N<30N<30` 运行规则 | 避免了小样本数据分析时的频繁误报骚扰   |
| **5. 级联失效**   | 第一步错，后面全错     | **【Gate 1】人机交互确认前置**                         | 确保后续分析的数据源 100% 正确         |

这种“AI 推导公式/解释业务 ➔ Python 负责计算/校验 ➔ 人类把守关键节点”的混合架构，在工程落地中能够提供远超纯 AI 方案的稳定性和准确率。
